## 一、 项目概述与战略定位

### 1.1 项目背景与核心痛点
中国县城及乡镇每年新增农村自建房数量庞大，但建筑设计环节长期处于"手绘草图+口头沟通"的原始状态。《川渝云图》项目旨在利用开源 3D 建筑编辑器 Pascal Editor 作为技术底座，打造面向中国下沉市场的农村自建房数字化平台。

根据对全球创业失败案例的系统性分析，**55%的创业公司死于产品与市场需求脱节（PMF幻觉），48%被竞争碾压，45%资金耗尽，35%单位经济模型崩溃** [1]。中国市场更为残酷——56%的失败公司因烧钱过度倒闭 [2]。因此，本项目彻底放弃向县城包工头收取软件订阅费的 SaaS 模式，转型为**"免费工具 + 交易抽佣"**的平台模式。

### 1.2 盈利模式全景（五大抽佣环节）
系统设计全面支撑以下五大盈利点，以单笔20万元自建房交易为例，平台综合收入约18,650元，综合抽佣率9.3%（略高于土巴兔8.0%、齐家网7.5%，处于行业合理区间 [3]）。

| 抽佣环节 | 抽佣比例/单价 | 单笔20万交易收入 | 系统功能支撑 |
| :--- | :--- | :--- | :--- |
| 线索派单费 | 30-300元/条 | 450元 | 线索收集表单 + 分级系统 + B端抢单大厅 |
| 施工成交佣金 | 3%-5% | 6,000元 | 资金托管 + 节点验收支付 |
| 建材集采佣金 | 5%-12% | 9,600元 | BOM表生成 + 建材商城 + CPS分佣 |
| 设计方案分成 | 20%-30% | 500元 | 精品户型方案库 + 付费解锁 |
| 金融保障返佣 | 1%-3% | 2,400元 | 贷款/保险API对接 |

## 二、 用户角色与核心业务流转

### 2.1 核心用户角色定义
1. **C端业主 (Homeowner)**：准备在农村/县城建房的居民。对价格敏感，缺乏专业建筑知识，需要直观的3D效果图来决策，担心包工头跑路或施工质量差。
2. **B端包工头/施工队 (Contractor)**：县城本地的施工负责人。文化程度不高，极度务实，拒绝为"提高效率"的软件付费，但极度渴望获取"能带来订单的建房线索"。
3. **独立设计师 (Designer)**：提供专业图纸的从业者。希望通过平台出售图纸变现。
4. **建材供应商 (Supplier)**：本地建材经销商。希望通过平台获取大宗采购订单。

### 2.2 核心业务流转旅程 (User Journey)
1. **触达与裂变**：包工头使用PC端免费画出3D别墅 → 生成微信小程序码/链接分享给潜在业主。
2. **激发与留资**：业主在手机端3D全景漫游 → 产生建房冲动 → 点击"获取本地施工报价" → 填写姓名、电话、预算、地址（生成线索）。
3. **分发与抢单**：线索进入平台公海 → 平台自动脱敏并分级（基础/意向/精准） → 3家本地包工头在抢单大厅支付150元购买 → 解锁业主电话。
4. **撮合与托管**：包工头联系业主线下看地 → 双方达成意向 → 在平台签订电子合同（对接法大大/e签宝） → 业主将20万工程款打入平台微信支付分账/担保账户。
5. **施工与验收**：包工头完成地基 → 拍照上传 → 业主点击确认验收 → 平台自动释放20%工程款给包工头（扣除3%平台佣金）。

## 三、 现有技术底座与基础设施评估

经过对源码的深度审查，发现项目已经内置了完善的后端基础设施，**无需从零开发用户系统和云端存储**，而是应该在现有架构上进行扩展。

### 3.1 已有基础设施清单
- **数据库与存储**：已集成 **Supabase**（PostgreSQL）。`env.mjs` 中已配置 `POSTGRES_URL` 和 `NEXT_PUBLIC_SUPABASE_URL`。
- **身份认证**：已集成 **Better Auth** 框架，支持 Google OAuth 和 Email Magic Link。
- **云端项目架构**：`Editor` 组件已原生支持 `projectId`，具备 `onLoad`、`onSave`、`onDirty` 回调机制。
- **自动保存与持久化**：已实现 `useAutoSave` Hook，本地使用 IndexedDB + Zundo 进行状态管理和撤销/重做。
- **项目可见性控制**：`SettingsPanel` 已实现 `isPrivate`、`showScansPublic`、`showGuidesPublic` 的权限控制。
- **缩略图生成**：已内置 `ThumbnailGenerator`，支持云端项目自动生成封面。

### 3.2 基础设施改造策略 (V1.0)
1. **认证系统扩展**：在 Better Auth 基础上，移除 Google OAuth，新增**微信扫码登录**（PC端）、**微信授权登录**（H5端）和**手机号验证码登录**（对接阿里云短信）。
2. **云端存储适配**：复用现有的 `onSave` 回调，将 SceneGraph JSON 数据直接存入 Supabase 的 `projects` 表中，利用 Supabase Storage 存储生成的缩略图和导出的 GLB 模型。
3. **邮件服务替换**：将现有的 Resend 邮件服务替换为国内的阿里云邮件推送或腾讯云 SES，用于系统通知。

## 四、 核心功能改造与本地化 (V1.0)

### 4.1 全面汉化与快捷键重映射
源码中存在大量硬编码的英文文案，必须通过引入 `next-intl` 进行全面替换。以下为核心汉化清单：

| 模块 | 英文原文 | 中文翻译 |
| :--- | :--- | :--- |
| **操作模式** | Select / Edit / Build / Delete | 选择 / 编辑 / 建造 / 删除 |
| **视图控制** | Perspective / Orthographic | 透视 / 正交 |
| **视图控制** | Stacked / Exploded / Solo / Manual | 堆叠 / 展开 / 独立 / 手动 |
| **工具栏** | Wall / Slab / Ceiling / Roof / Door / Window | 墙体 / 楼板 / 天花板 / 屋顶 / 门 / 窗户 |
| **家具分类** | Furniture / Appliance / Kitchen / Bathroom | 家具 / 电器 / 厨房 / 卫浴 |
| **面板属性** | Position / Dimensions / Frame / Swing / Threshold | 位置 / 尺寸 / 框架 / 开启方式 / 门槛 |
| **设置面板** | Visibility / Public / Export / Thumbnail | 可见性 / 公开 / 导出 / 缩略图 |
| **错误提示** | The editor scene failed to render | 编辑器场景渲染失败 |

**快捷键重映射**：
- 撤销/重做统一为符合国人习惯的 `Ctrl+Z` 和 `Ctrl+Y`（或 `Ctrl+Shift+Z`）。
- 删除操作统一支持 `Delete` 和 `Backspace`。

### 4.2 中式素材库建设 (catalog-items.tsx)
现有素材库全为西式家具，必须新增中式建筑与家具模型。
- **采购/制作标准**：单模型面数 < 10,000，贴图分辨率 < 1024x1024，格式必须为 GLB。
- **首批必上清单**：中式双坡/四坡屋顶、圆形/方形罗马柱、标准防盗门、铝合金推拉窗、中式大圆桌、堂屋神龛、土灶台、砖砌院墙。

### 4.3 地图服务替换与性能降级
- **地图服务**：移除 Google Maps API，接入**高德地图 Web JS API**。在 `site-panel` 中实现基于高德的地址搜索、经纬度获取和卫星图切片加载。
- **性能降级 (Plan B)**：在 `packages/viewer/src/components/viewer/index.tsx` 初始化时检测 WebGL 上下文。若创建失败或连续 5 秒帧率低于 15fps，自动卸载 `<Canvas>` 组件，替换为纯 2D 的 `FloorplanPanel`（平面图）或静态渲染图片轮播。

## 五、 数据库设计与 API 规范

### 5.1 核心数据库 ER 图设计 (Supabase PostgreSQL)

**1. users (用户表)**
- `id` (UUID, PK)
- `phone` (VARCHAR(11), UNIQUE)
- `wechat_openid` (VARCHAR(64), UNIQUE)
- `role` (ENUM: 'owner', 'contractor', 'designer')
- `credit_score` (INT, default: 100) - 信用分

**2. projects (项目表)**
- `id` (UUID, PK)
- `user_id` (UUID, FK -> users.id)
- `title` (VARCHAR(64))
- `scene_data` (JSONB) - 存储完整的 SceneGraph
- `thumbnail_url` (VARCHAR(256))
- `share_code` (VARCHAR(8), UNIQUE)
- `is_private` (BOOLEAN, default: false)

**3. leads (线索表)**
- `id` (UUID, PK)
- `owner_id` (UUID, FK -> users.id)
- `project_id` (UUID, FK -> projects.id, NULLABLE)
- `address_code` (VARCHAR(12)) - 乡镇级行政区划代码
- `budget_level` (INT)
- `status` (ENUM: 'open', 'sold_out', 'closed')
- `max_sales` (INT, default: 3)
- `current_sales` (INT, default: 0)
- `expires_at` (TIMESTAMP) - 7天降价，30天下架

**4. lead_orders (线索订单表)**
- `id` (UUID, PK)
- `lead_id` (UUID, FK -> leads.id)
- `contractor_id` (UUID, FK -> users.id)
- `amount` (DECIMAL(10,2))
- `status` (ENUM: 'pending', 'paid', 'refunded', 'appealing')

### 5.2 核心 API 接口规范 (RESTful)

| 接口路径 | 方法 | 描述 | 请求体示例 | 响应体示例 |
| :--- | :--- | :--- | :--- | :--- |
| `/api/auth/wechat/login` | POST | 微信登录 | `{ code: "wx_code" }` | `{ token: "jwt_token", user: {...} }` |
| `/api/projects/:id/sync` | POST | 同步场景数据 | `{ scene_data: {...} }` | `{ success: true, updated_at: "..." }` |
| `/api/leads` | POST | C端提交线索 | `{ phone: "...", address: "...", budget: 200000 }` | `{ lead_id: "..." }` |
| `/api/leads/market` | GET | B端获取线索大厅列表 | `?region=500100` | `[{ id: "...", budget: "20-30万", ... }]` (脱敏) |
| `/api/leads/:id/buy` | POST | B端购买线索 | `{}` | `{ order_id: "...", pay_url: "..." }` |

## 六、 版本迭代路线图 (Roadmap)

### 6.1 V1.0 极简MVP版（第1-3个月）
**核心任务**：低成本改造现有源码，跑通"工具→分享→留资→卖线索"的最小闭环。
- **功能**：全面汉化、中式素材库、微信/手机号登录、高德地图接入、H5 3D预览页、C端留资表单、B端抢单大厅。
- **验证标准**：创始人亲自带着平板电脑去县城建材市场，连续拜访50个包工头。如果有超过10%（5人）愿意支付150元购买线索，则PMF初步成立。

### 6.2 V2.0 商业闭环版（第4-8个月）
**核心任务**：跑通"线索→签约→施工→验收→付款"的完整交易闭环，防止飞单。
- **功能**：对接微信支付"电商收付通"实现资金托管与节点验收、对接电子签章API、包工头信用分与评价系统、精品户型方案库（付费解锁图纸）。
- **防飞单机制**：向业主强调"资金托管"的安全价值，免费赠送"施工意外险"（对接众安保险API）。

### 6.3 V3.0 供应链集采版（第9-15个月）
**核心任务**：接入建材供应链，实现"建材集采佣金"这一最大现金牛。
- **功能**：基于 SceneGraph 自动生成 BOM 表（精确计算砖块、混凝土用量，含5%损耗系数）、建材供应商入驻后台、集采询价与CPS分佣系统。
- **AI壁垒**：引入大语言模型（如 DeepSeek-V3 或 阿里云通义千问，API调用成本低），实现"自然语言一键生成农村别墅"功能，建立技术护城河。

## 七、 运营策略与增长飞轮

### 7.1 冷启动策略 (获取首批50个包工头)
- **地推话术**：绝口不提"3D建模软件"，直接展示H5预览页，话术为："张老板，我们平台上有你们镇上想建房的业主电话，你要不要花150块钱买一个去联系看看？"
- **演示流程**：用平板展示本地真实的宅基地3D方案，证明线索的真实性。
- **激励方案**：首批入驻的50个包工头，免费赠送3条基础线索，完成首单交易免收3%平台佣金。

### 7.2 微信生态裂变路径
1. **工具赋能**：包工头使用PC端设计方案，点击"分享"生成带有自己联系方式的**微信小程序码**。
2. **社交传播**：包工头将小程序码发到朋友圈或业主微信群，配文"今年最火的3层别墅款式，点击查看3D全景"。
3. **C端留资**：潜在业主扫码进入小程序，被3D效果震撼，点击底部"获取本地施工报价"留下电话。
4. **线索回流**：该线索优先（30分钟内）免费推送给分享该方案的包工头；若其放弃，则流入公海大厅供其他包工头购买。

### 7.3 关键事件埋点
- `project_created`：创建项目
- `share_link_generated`：生成分享链接
- `h5_viewed`：H5预览页浏览（记录停留时长）
- `lead_submitted`：提交线索
- `lead_purchased`：购买线索

## 八、 财务模型与成本估算 (V1.0阶段)

为规避"资金耗尽"风险，V1.0 阶段必须将月度消耗控制在 5 万元以内。

### 8.1 研发与运营成本 (月度)
- **团队薪资**：创始人(0元) + 1名全栈工程师(1.5万) + 1名前端工程师(1.2万) = 2.7万元
- **外包成本**：中式3D模型定制（约20个，单价500元）= 1.0万元（一次性）
- **差旅地推**：县城交通与住宿 = 0.5万元

### 8.2 基础设施成本 (月度)
- **服务器 (阿里云)**：2核4G ECS (约150元) + Supabase 托管版 (Pro Plan $29/约200元) = 350元
- **对象存储 (OSS)**：500GB 存储包 + 100GB 流量包 = 约 100元
- **短信服务**：阿里云短信（0.04元/条，按1万条计）= 400元
- **地图API**：高德地图企业版（初期免费额度够用）= 0元

### 8.3 合规与认证成本 (一次性)
- **微信开放平台认证**：300元/年
- **域名注册与SSL证书**：约 200元/年
- **ICP备案与公安联网备案**：免费（需耗时约20天）

## 九、 失败规避策略矩阵与 Pivot 机制

### 9.1 失败规避矩阵
| 失败原型 | 加权占比 | 典型案例 | 在本项目中的表现 | 系统级规避策略 |
| :--- | :--- | :--- | :--- | :--- |
| **PMF幻觉** | 55% | Quibi | 包工头不需要3D建模 | V1.0极简MVP+50人地推验证；工具免费，卖线索而非卖工具 |
| **竞争碾压** | 48% | Jawbone | 酷家乐推出免费下沉版 | 不拼渲染精度，拼本地化交易闭环（资金托管+建材集采） |
| **资金耗尽** | 45% | 同程生活 | 过早组建大团队 | V1.0仅3人团队，月消耗<5万；严格执行Pivot机制 |
| **单位经济崩溃** | 35% | WeWork | CAC远高于LTV | 微信社交裂变降CAC至<100元；线索单价150元+施工佣金6000元 |

### 9.2 严格的 Pivot (转型) 机制
每个版本上线后，必须在规定时间内达到硬性指标。**未达标则必须立即启动Pivot，绝不盲目烧钱**。

| 版本 | 验收时间 | 必须达标指标 | 未达标应对 (Pivot方向) |
| :--- | :--- | :--- | :--- |
| V1.0 | 上线后1个月 | 至少5个包工头愿意付费购买线索 | 立即Pivot：放弃B端收费，转型为纯C端"建房灵感社区"赚取广告费 |
| V2.0 | 上线后2个月 | 至少产生3笔通过平台资金托管的施工交易 | 降低抽佣比例至1%，或放弃交易闭环，退回纯线索分发平台 |
| V3.0 | 上线后3个月 | 建材集采GMV突破50万元/月 | 放弃自建供应链，转为对接京东/淘宝建材赚取淘宝客佣金 |
